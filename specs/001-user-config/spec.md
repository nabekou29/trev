# Feature Specification: TOML + Lua ユーザー設定

**Feature Branch**: `001-user-config`
**Created**: 2026-02-10
**Status**: Draft
**Input**: User description: "TOMLとluaによる user configuration"

## User Scenarios & Testing

### User Story 1 - TOML による静的設定 (Priority: P1)

ユーザーは TOML ファイルを編集して trev の表示・ソート・テーマなどの設定を変更できる。設定ファイルが存在しない場合はデフォルト値で動作し、構文エラーがある場合はわかりやすいエラーメッセージが表示される。

**Why this priority**: 設定の基盤。他のすべての設定機能がこの仕組みに依存する。

**Independent Test**: TOML ファイルを配置し、trev を起動して設定が反映されていることを目視確認できる。

**Acceptance Scenarios**:

1. **Given** 設定ファイルが存在しない, **When** trev を起動する, **Then** すべてデフォルト値で動作する
2. **Given** `[sort] order = "size"` が設定されている, **When** trev を起動する, **Then** ファイルがサイズ順で表示される
3. **Given** 設定ファイルに構文エラーがある, **When** trev を起動する, **Then** エラー箇所と行番号を含むメッセージが表示され、デフォルト設定で起動する
4. **Given** 未知の設定キーがある, **When** trev を起動する, **Then** 警告を出しつつ正常に起動する

---

### User Story 2 - Lua による動的設定・カスタマイズ (Priority: P2)

ユーザーは Lua スクリプトで条件分岐やカスタムロジックを含む設定を記述できる。Lua 設定は TOML 設定を上書きでき、キーバインディングのカスタマイズやカスタムアクションの定義が可能。

**Why this priority**: パワーユーザー向けの拡張性。TOML だけではカバーできない動的な設定ニーズに対応する。

**Independent Test**: Lua 設定ファイルを配置し、カスタムキーバインドが動作することを確認できる。

**Acceptance Scenarios**:

1. **Given** Lua 設定ファイルが存在する, **When** trev を起動する, **Then** Lua で定義した設定が TOML 設定を上書きして適用される
2. **Given** Lua でカスタムキーバインドを定義, **When** そのキーを押す, **Then** 定義したアクションが実行される
3. **Given** Lua スクリプトにエラーがある, **When** trev を起動する, **Then** エラー内容と行番号が表示され、Lua 設定なしで起動する（TOML 設定は有効）

---

### User Story 3 - CLI 引数による設定の上書き (Priority: P3)

ユーザーは CLI 引数で設定値を一時的に上書きできる。設定の優先順位は CLI > Lua > TOML > デフォルト。

**Why this priority**: 一時的な設定変更のニーズ。スクリプトからの利用にも便利。

**Independent Test**: `trev -a` で起動し、TOML で `show_hidden = false` でも隠しファイルが表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** TOML で `show_hidden = false`, **When** `trev -a` で起動, **Then** 隠しファイルが表示される
2. **Given** Lua で `sort.order = "size"`, **When** CLI でソート順を指定, **Then** CLI の指定が優先される

---

### Edge Cases

- 設定ファイルのパーミッションが読み取り不可の場合 → エラーメッセージを表示し、デフォルト設定で起動
- 設定ファイルが空の場合 → すべてデフォルト値として扱う
- TOML と Lua の両方が存在する場合の読み込み順序 → TOML を先に読み、Lua で上書き
- Lua スクリプトが無限ループする場合 → 実行時間に上限を設け、タイムアウト時は警告を出して Lua 設定をスキップ
- 設定ファイルのホットリロード → v1.0 ではスコープ外。起動時のみ読み込み

## Requirements

### Functional Requirements

- **FR-001**: 設定ファイルの探索パスは `$XDG_CONFIG_HOME/trev/` を最優先とし、存在しない場合は `~/.config/trev/` をフォールバックとする
- **FR-002**: TOML 設定ファイル (`config.toml`) で以下のセクションを定義できる: `sort`, `display`, `theme`, `keymap`
- **FR-003**: Lua 設定ファイル (`init.lua`) で TOML と同等の設定を定義でき、加えてカスタムキーバインドとカスタムアクションを定義できる
- **FR-004**: 設定の適用優先順位は CLI引数 > Lua > TOML > デフォルト値
- **FR-005**: 設定ファイルの構文エラーは、エラー箇所（ファイル名・行番号）を含む明確なメッセージで報告する
- **FR-006**: 未知の設定キーは警告として報告するが、アプリケーションの起動を妨げない
- **FR-007**: Lua スクリプトの実行にはタイムアウト（上限 3 秒）を設け、超過時は警告を出してスキップする
- **FR-008**: 設定可能な項目は以下を含む:
  - ソート: 順序・方向・ディレクトリ優先
  - 表示: 隠しファイル・gitignore・プレビュー・アイコン・メタ情報・プレビュー幅比率
  - テーマ: テーマ名・シンタックスハイライトテーマ
  - キーマップ: プリセット (vim / emacs / custom) およびカスタムバインド
- **FR-009**: Lua API は表示設定・キーバインド・カスタムアクション定義に限定し、ファイル操作 API は提供しない。ただし、カスタムアクションの callback は選択中のアイテム情報（パス、メタデータ等）やツリー状態を引数として受け取れるようにし、ユーザーが外部コマンド呼び出し等で自由に拡張できるようにする

### Key Entities

- **Config**: アプリケーション全体の設定を保持。TOML/Lua/CLI の各ソースからマージされる
- **KeyBinding**: キー入力とアクションのマッピング。プリセット + カスタム定義で構成
- **Theme**: UI の色・スタイル定義。ビルトインテーマ + ユーザー定義

## Success Criteria

### Measurable Outcomes

- **SC-001**: 設定ファイルなしの状態から、ユーザーが TOML 設定を作成して反映を確認するまで 5 分以内で完了できる
- **SC-002**: 構文エラーのある設定ファイルに対して、エラーメッセージにファイル名と行番号が 100% 含まれている
- **SC-003**: 設定ファイルの読み込みがアプリケーション起動時間に追加する遅延は 50ms 以下
- **SC-004**: Lua によるカスタムキーバインドの定義が 10 行以内のコードで実現可能
- **SC-005**: 未知の設定キーや軽微なエラーでアプリケーションがクラッシュするケースがゼロ

## Assumptions

- 設定ファイルのホットリロード（起動中の変更反映）は v1.0 スコープ外
- Lua ランタイムは組み込み型を使用し、外部の Lua インストールを要求しない
- テーマのカスタマイズはカラー定義のみで、レイアウト変更は含まない
- XDG Base Directory Specification に準拠する

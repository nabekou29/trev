# Feature Specification: File Operations + FS Change Detection + Undo/Redo + Session Persistence

**Feature Branch**: `012-file-operations`
**Created**: 2026-02-15
**Status**: Draft
**Input**: ファイル操作（コピー、移動、削除、作成、リネーム）+ FS変更検出 + Undo/Redo + セッション永続化

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ファイルのコピー・移動 (Priority: P1)

ユーザーは trev 内で複数のファイルやディレクトリを別の場所にコピーまたは移動できる。vim ライクな yank/paste モデル（`y`=コピー、`x`=カット、`p`=ペースト）を使い、直感的に操作できる。複数ファイルを対象にする場合は `Space` でマーク（複数選択）してからまとめて操作する。

**Why this priority**: コピーと移動はファイルマネージャーの最も基本的な操作であり、他のすべてのファイル操作の土台となる。yank/paste モデルが他の操作（削除の undo を含む）の基盤にもなる。

**Independent Test**: テスト用ディレクトリ内でファイルを yank → 別ディレクトリに paste し、コピー/移動が正しく完了することを確認する。

**Acceptance Scenarios**:

1. **Given** ファイルにカーソルがある状態, **When** `y` → 目的ディレクトリに移動 → `p`, **Then** ファイルがコピーされ、元のファイルも存在する
2. **Given** ファイルにカーソルがある状態, **When** `x` → 目的ディレクトリに移動 → `p`, **Then** ファイルが移動され、元の場所からは消える
3. **Given** 3ファイルを `Space` でマーク済み, **When** `y` → 目的ディレクトリに移動 → `p`, **Then** 3ファイルすべてがコピーされる
4. **Given** 同名ファイルが存在する場所にペースト, **When** `p`, **Then** 自動リネーム（`file_1.txt` 形式）で衝突を回避して貼り付けられる
5. **Given** コピー後のバッファ, **When** 別のディレクトリに再度 `p`, **Then** 再度ペーストできる（バッファは保持される）
6. **Given** カット後のバッファ, **When** `p` 実行後, **Then** バッファはクリアされる（移動は1回のみ）

---

### User Story 2 - ファイル・ディレクトリの新規作成とリネーム (Priority: P1)

ユーザーは trev 内でファイルやディレクトリを新規作成したり、既存のファイルをリネームできる。操作はツリービュー内のインライン入力フィールドで行い、エディタから離れることなく完結する。

**Why this priority**: 新規作成とリネームはコピー・移動と並ぶ基本操作であり、日常のファイル管理に不可欠。

**Independent Test**: `a` でインライン入力を開き、ファイル名を入力して Enter → ファイルが作成されることを確認する。

**Acceptance Scenarios**:

1. **Given** ディレクトリにカーソルがある状態, **When** `a` → `newfile.txt` と入力 → Enter, **Then** そのディレクトリ内に `newfile.txt` が作成される
2. **Given** ファイルにカーソルがある状態, **When** `a` → `newfile.txt` と入力 → Enter, **Then** 親ディレクトリ内に `newfile.txt` が作成される
3. **Given** `a` で入力中, **When** 末尾に `/` をつけて入力 → Enter, **Then** ディレクトリが作成される
4. **Given** `a` で入力中, **When** `foo/bar/baz.txt` と入力 → Enter, **Then** 中間ディレクトリも含めて作成される
5. **Given** ファイルにカーソルがある状態, **When** `r` → 既存名が入力フィールドに表示され、カーソルは拡張子の前に配置される
6. **Given** リネーム中, **When** 新しい名前を入力 → Enter, **Then** ファイルがリネームされる
7. **Given** 入力中, **When** Esc, **Then** 操作がキャンセルされ元の状態に戻る

---

### User Story 3 - ファイルの削除 (Priority: P1)

ユーザーはファイルやディレクトリを削除できる。2つの削除方法があり、設定で `d` キーの動作を選択できる。

- `d`: 完全削除（デフォルト）または独自 trash 経由の削除（設定による）
- `D`: 常にOS標準ゴミ箱に送る

削除操作は確認ダイアログで意図しない削除を防止する。

**Why this priority**: 削除はファイル管理の基本操作であり、安全性（確認ダイアログ、undo 可能な削除モード）がユーザーの信頼に直結する。

**Independent Test**: ファイルに対して `d` → 確認ダイアログで `y` → ファイルが削除されることを確認する。

**Acceptance Scenarios**:

1. **Given** `delete_mode = "permanent"` の設定でファイルにカーソル, **When** `d`, **Then** 確認ダイアログ「Permanently delete 'filename'?」が表示される
2. **Given** 確認ダイアログ表示中, **When** `y` or Enter, **Then** ファイルが完全削除される（undo 不可）
3. **Given** 確認ダイアログ表示中, **When** `n` or Esc, **Then** 削除がキャンセルされファイルは残る
4. **Given** `delete_mode = "custom_trash"` の設定でファイルにカーソル, **When** `d` → 確認(`y`), **Then** ファイルが独自 trash に移動される（undo 可能）
5. **Given** ファイルにカーソル, **When** `D` → 確認(`y`), **Then** ファイルがOS標準ゴミ箱に送られる（undo 不可）
6. **Given** 3ファイルをマーク済み, **When** `d`, **Then** 確認ダイアログ「Permanently delete 3 files?」と対象ファイルリストが表示される

---

### User Story 4 - Undo / Redo (Priority: P2)

ユーザーはファイル操作を取り消し（undo）たり、やり直し（redo）たりできる。操作履歴はセッションをまたいで永続化される。undo/redo 前にファイルシステムの状態を検証し、不整合がある場合は安全に中止する。

**Why this priority**: undo/redo はファイル操作の安全性を大幅に向上させる重要機能だが、基本的なファイル操作（P1）が先に動作する必要がある。

**Independent Test**: ファイルをコピー → `u` で undo → コピー先が削除されることを確認する。`Ctrl+r` で redo → 再度コピーされることを確認する。

**Acceptance Scenarios**:

1. **Given** ファイルをコピーした直後, **When** `u`, **Then** コピーされたファイルが削除される（undo）
2. **Given** undo した直後, **When** `Ctrl+r`, **Then** ファイルが再度コピーされる（redo）
3. **Given** `delete_mode = "custom_trash"` で削除した直後, **When** `u`, **Then** ファイルが元の場所に復元される
4. **Given** 3ファイルのコピーをまとめて行った直後, **When** `u`, **Then** 3ファイルすべてのコピーがまとめて取り消される（グループ化）
5. **Given** undo 後にファイルが外部から変更された, **When** `Ctrl+r`, **Then** 「Cannot redo: file state has changed」と表示され、redo は中止される
6. **Given** 完全削除（permanent）を実行した直後, **When** `u`, **Then** 何も起こらない（永久削除は undo 不可）
7. **Given** 前回セッションでファイル操作を行った, **When** 今回セッションで起動 → `u`, **Then** 前回の操作が undo される（履歴が永続化されている）

---

### User Story 5 - FS 変更検出 (Priority: P2)

展開中のディレクトリ内でファイルの追加・削除・変更が外部から行われた場合、trev のツリー表示が自動的に更新される。折り畳み中のディレクトリで変更が検出された場合は、再展開時に最新の状態が表示される。

**Why this priority**: ファイル操作を行う以上、外部変更との整合性が必要。ただし P1 のファイル操作なしには意味をなさない。

**Independent Test**: trev で展開中のディレクトリ内に外部からファイルを追加し、ツリー表示が自動更新されることを確認する。

**Acceptance Scenarios**:

1. **Given** 展開済みディレクトリを表示中, **When** 外部からファイルが追加される, **Then** 250ms のデバウンス後にツリーに反映される
2. **Given** 展開済みディレクトリを表示中, **When** 外部からファイルが削除される, **Then** ツリーから該当ノードが消える
3. **Given** trev 自身がファイル操作中, **When** その操作に起因する FS イベントが発生, **Then** 重複した更新は行われない（自己操作を区別する）
4. **Given** 折り畳み中のディレクトリで外部変更が発生, **When** そのディレクトリを再展開, **Then** 最新のファイル一覧が表示される（古いキャッシュではない）

---

### User Story 6 - セッション永続化 (Priority: P3)

ユーザーが trev を再起動した際に、前回のセッション状態（展開状態、カーソル位置、マーク、yank バッファ、undo 履歴）が復元される。復元の有無は CLI オプションおよび設定ファイルで制御できる。

**Why this priority**: 利便性を向上させる機能だが、ファイル操作（P1）と undo/redo（P2）が先に完成している必要がある。

**Independent Test**: trev でディレクトリを展開 → 終了 → 再起動 → 展開状態が復元されることを確認する。

**Acceptance Scenarios**:

1. **Given** ディレクトリを展開しカーソルを移動した状態で終了, **When** 同じルートパスで再起動, **Then** 展開状態とカーソル位置が復元される
2. **Given** `--no-restore` で起動, **When** セッションファイルが存在しても, **Then** 初期状態で起動する（セッションファイルは削除されない）
3. **Given** `--restore` で起動, **When** セッションファイルが存在, **Then** 前回の状態が復元される
4. **Given** 復元時に展開ディレクトリの一部が外部から削除済み, **When** 復元, **Then** 存在しないパスは除外されて残りの状態だけが復元される
5. **Given** `session.restore_by_default = true` の設定, **When** CLI オプションなしで起動, **Then** 自動的にセッションが復元される
6. **Given** 90日以上アクセスのないセッションファイル, **When** 起動時, **Then** 古いセッションファイルが自動削除される

---

### User Story 7 - マーク（複数選択） (Priority: P1)

ユーザーは `Space` でファイルやディレクトリをマークし、複数のアイテムをまとめてファイル操作の対象にできる。マーク状態はツリー表示で視覚的に区別される。

**Why this priority**: 複数選択は yank/paste や削除の対象指定に不可欠な基盤機能。

**Independent Test**: `Space` でファイルをマーク → 視覚的に区別されることを確認 → `y` → マーク済みファイルが yank バッファに入ることを確認する。

**Acceptance Scenarios**:

1. **Given** ファイルにカーソルがある状態, **When** `Space`, **Then** ファイルがマークされ視覚的に区別され、カーソルが1つ下に移動する
2. **Given** マーク済みファイルにカーソルがある状態, **When** `Space`, **Then** マークが解除される
3. **Given** 3ファイルをマーク済み, **When** `y`, **Then** マーク済みの3ファイルが yank バッファに入り、マークはクリアされる
4. **Given** マークなし, **When** `y`, **Then** カーソル位置のファイルが yank バッファに入る

---

### User Story 8 - UI フィードバック (Priority: P2)

ユーザーはファイル操作の状態や結果をステータスバーで確認できる。yank バッファの状態、マーク数、操作結果が表示される。確認ダイアログは画面中央にモーダル表示される。

**Why this priority**: 操作の可視性はユーザー体験に重要だが、操作自体が動作してから磨く機能。

**Independent Test**: `y` でファイルを yank → ステータスバーに `[Y:1]` が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 3ファイルをコピー yank した状態, **When** ステータスバーを確認, **Then** `[Y:3]` と表示される
2. **Given** 1ファイルをカット yank した状態, **When** ステータスバーを確認, **Then** `[X:1]` と表示される
3. **Given** 5ファイルをマーク済み, **When** ステータスバーを確認, **Then** `[M:5]` と表示される
4. **Given** ファイルをコピーした直後, **When** ステータスバーを確認, **Then** `Copied 3 files` が一時表示される（3秒後に消える）
5. **Given** undo 実行直後, **When** ステータスバーを確認, **Then** `Undone: Copy 3 files` が一時表示される

---

### Edge Cases

- 読み取り専用ディレクトリへの書き込み操作を試みた場合、適切なエラーメッセージが表示される
- ペースト先のディスク容量が不足している場合、操作が安全に中断される
- undo スタックが最大サイズに達した場合、最古のエントリが削除され、関連する独自 trash ファイルも完全削除される
- 非常に深いネストのディレクトリ構造をペーストした場合でも正しく動作する
- シンボリックリンクはリンク自体を操作対象とする（コピー=リンクを複製、移動=リンクを移動、削除=リンクのみ削除。リンク先は辿らない）
- 自動リネーム時に極端に多い同名ファイルが存在する場合でも一意な名前が生成される
- セッション復元時にルートパスが変更されている場合は新規セッションとして起動する
- FS 変更検出で大量の変更イベントが同時に発生した場合、デバウンスにより適切にバッチ処理される
- ディレクトリを自身の子孫にコピー/移動しようとした場合、操作前に検出してエラー表示で中止する（「Cannot copy a directory into itself」）
- カット yank 後にペースト前にソースファイルが外部削除された場合、エラーメッセージが表示される
- インライン入力中にファイル名に使用不可の文字が入力された場合のハンドリング

## Requirements *(mandatory)*

### Functional Requirements

**コピー・移動**:
- **FR-001**: ユーザーは `y` でカーソル位置またはマーク済みファイルを yank バッファにコピー登録できなければならない
- **FR-002**: ユーザーは `x` でカーソル位置またはマーク済みファイルを yank バッファにカット登録できなければならない
- **FR-003**: ユーザーは `p` で yank バッファの内容をカーソル位置のディレクトリにペーストできなければならない
- **FR-004**: ペースト時に同名ファイルが存在する場合、自動リネーム（`file_1.txt` 形式）で衝突を回避しなければならない
- **FR-005**: コピー yank 後のバッファは保持され、複数回ペーストできなければならない
- **FR-006**: カット yank 後はペースト実行後にバッファがクリアされなければならない

**新規作成・リネーム**:
- **FR-007**: ユーザーは `a` でインライン入力を開き、ファイルまたはディレクトリを新規作成できなければならない
- **FR-008**: 入力値の末尾が `/` の場合はディレクトリ、それ以外はファイルとして作成しなければならない
- **FR-009**: 中間ディレクトリ（`foo/bar/baz.txt`）が存在しない場合は自動的に作成しなければならない
- **FR-010**: ユーザーは `r` でカーソル位置のファイルをインライン入力でリネームできなければならない
- **FR-011**: リネーム開始時、入力フィールドに既存名が入り、カーソルは拡張子の前に配置されなければならない

**削除**:
- **FR-012**: `d` キーの動作は設定（`delete_mode`）に依存しなければならない: `"permanent"`（デフォルト）= 完全削除、`"custom_trash"` = 独自 trash へ移動
- **FR-013**: `D` キーは常にOS標準ゴミ箱にファイルを送らなければならない
- **FR-014**: 削除操作（`d` および `D`）は実行前に確認ダイアログを表示しなければならない
- **FR-015**: 独自 trash は専用ディレクトリにタイムスタンプ付きファイル名で保存しなければならない

**マーク（複数選択）**:
- **FR-016**: ユーザーは `Space` でファイル/ディレクトリのマークをトグルでき、カーソルが1つ下に移動しなければならない
- **FR-017**: マーク済みファイルがある場合、ファイル操作はマーク済みファイルを対象としなければならない
- **FR-018**: マーク済みファイルがない場合、ファイル操作はカーソル位置のファイルを対象としなければならない
- **FR-019**: yank または delete の実行後、マークはクリアされなければならない
- **FR-020**: マーク済みファイルはツリー表示で視覚的に区別されなければならない

**Undo/Redo**:
- **FR-021**: ユーザーは `u` で直前の操作を undo できなければならない（undo 可能な操作に限る）
- **FR-022**: ユーザーは `Ctrl+r` で undo した操作を redo できなければならない
- **FR-023**: 複数ファイルの操作は1つのグループとしてまとめて undo/redo されなければならない
- **FR-024**: undo/redo 実行前にファイルシステムの状態を検証し、不整合がある場合は中止してエラーを表示しなければならない
- **FR-025**: 完全削除（permanent delete）とOS標準ゴミ箱への移動は undo 不可としなければならない
- **FR-026**: undo 不可な操作実行後、「This operation cannot be undone」と一時表示しなければならない
- **FR-027**: undo 履歴のスタックサイズは設定可能で、溢れた場合は最古のエントリから削除しなければならない
- **FR-028**: 独自 trash を参照する OpGroup がスタックから溢れた場合、対応する trash ファイルを完全削除しなければならない

**FS 変更検出**:
- **FR-029**: 展開済みディレクトリを監視し、外部からのファイル追加・削除・変更をツリーに自動反映しなければならない
- **FR-030**: FS イベントは 250ms のデバウンスでバッチ処理しなければならない
- **FR-031**: trev 自身のファイル操作中は監視を一時停止し、操作完了後に再開しなければならない
- **FR-032**: 折り畳み中のディレクトリで変更が検出された場合、再展開時に最新の内容を表示しなければならない

**セッション永続化**:
- **FR-033**: ルートパスごとにセッション状態（展開状態、カーソル位置、マーク、yank バッファ、undo 履歴）を保存しなければならない
- **FR-034**: 起動時に `--restore` / `--no-restore` で復元の有無を制御できなければならない
- **FR-035**: CLI オプション未指定時は `session.restore_by_default` 設定に従わなければならない
- **FR-036**: 復元時に存在しないパスへの参照は自動的に除外されなければならない
- **FR-037**: 設定期間（デフォルト90日）以上アクセスのないセッションファイルは起動時に自動削除しなければならない

**UI**:
- **FR-038**: インライン入力は基本的なテキスト編集（文字入力、BackSpace、Delete、Home/End、左右矢印、Ctrl+U）をサポートしなければならない
- **FR-039**: 確認ダイアログは画面中央にモーダル表示され、操作対象のファイルリストを含まなければならない
- **FR-040**: ステータスバーに yank 状態（`[Y:N]` / `[X:N]`）、マーク数（`[M:N]`）、操作結果を表示しなければならない
- **FR-041**: 操作結果の一時表示は3秒後に自動的に消えなければならない
- **FR-043**: ファイル操作の実行中はユーザー入力をブロックし、ステータスバーに「Processing...」と表示しなければならない

**設定**:
- **FR-042**: `delete_mode`、`undo_stack_size`（デフォルト100）、`restore_by_default`（デフォルト true）、`expiry_days`（デフォルト90）、`watcher.enabled`（デフォルト true）、`watcher.debounce_ms`（デフォルト250）を設定ファイルで制御できなければならない

### Key Entities

- **Yank バッファ**: 対象ファイルパスのリストと操作モード（コピーまたはカット）を保持する。yank 操作で上書きされ、ペースト操作の入力となる
- **マーク**: ユーザーが選択した複数ファイル/ディレクトリの集合。ファイル操作の対象を決定するために使用される
- **操作グループ (OpGroup)**: 1つのユーザー操作に含まれる複数のファイルシステム変更をまとめたもの。undo/redo の最小単位。正方向の操作リスト、逆方向の操作リスト、事前検証条件を含む
- **Undo 履歴**: 操作グループのスタック。カーソル位置により undo/redo の対象が決まる。永続化対象
- **独自 trash**: 設定時に `d` で削除されたファイルの一時保管先。undo によって元の位置に復元可能。タイムスタンプ付きファイル名で衝突を回避
- **セッション状態**: ルートパスごとに保存される trev の状態スナップショット。展開状態、カーソル、マーク、yank、undo 履歴を含む
- **インライン入力**: ツリービュー内に表示される1行のテキスト入力フィールド。新規作成やリネーム操作に使用
- **確認ダイアログ**: 画面中央のモーダルウィジェット。削除操作の確認に使用

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: すべてのファイル操作（コピー、移動、削除、作成、リネーム）が1秒以内に完了する（100ファイル以下の場合）
- **SC-002**: undo/redo 操作が1秒以内に完了する
- **SC-003**: FS 変更がツリーに反映されるまでの遅延が500ms 以内である（デバウンス 250ms + 処理時間）
- **SC-004**: セッション復元が2秒以内に完了する（1000ノード以下のツリーの場合）
- **SC-005**: undo 履歴が100エントリに達しても操作のレスポンスに影響がない
- **SC-006**: undo 可能な操作について、undo → redo のラウンドトリップでファイルシステムが元の状態に完全に戻る
- **SC-007**: 完全削除（permanent）とOS標準ゴミ箱の操作が undo スタックに入らず、「undo 不可」のフィードバックが表示される
- **SC-008**: 外部からの同時多数のファイル変更（100ファイル以上）でもアプリケーションがクラッシュせず、最終的にツリーが正しい状態に収束する

## Clarifications

### Session 2026-02-15

- Q: シンボリックリンクのファイル操作時の扱いは？ → A: リンク自体を操作対象とする（コピー=リンクを複製、移動=リンクを移動、削除=リンクのみ削除。リンク先は辿らない）
- Q: 長時間のファイル操作時のUXは？ → A: ブロッキング（操作中は入力不可、ステータスバーに「Processing...」表示）
- Q: クラッシュ時の部分操作の扱いは？ → A: 放置（コピー済みファイルは残る、undo 履歴は最後の保存時点のまま。ユーザーが手動対応）
- Q: ディレクトリの自己参照コピー/移動の防止は？ → A: 操作前に検出してエラー表示で中止（「Cannot copy a directory into itself」）

### Assumptions

- ファイル操作のパフォーマンス目標はローカルファイルシステムを前提とする（ネットワークドライブは対象外）
- OS標準ゴミ箱の操作は `trash` クレートの対応プラットフォーム（Linux、macOS、Windows）に依存する
- セッションファイルの保存先はプラットフォームのデータディレクトリ規約に従う（XDG on Linux、Library/Application Support on macOS）
- FS 変更検出はプラットフォームのネイティブ API（inotify、FSEvents、kqueue）を利用する
- 独自 trash のクリーンアップは undo スタックの溢れ時にのみ実行され、ユーザーによる明示的なクリーンアップ操作は提供しない
- クラッシュ時の部分操作（例: 10ファイル中5ファイルのみコピー完了）はロールバックしない。コピー済みファイルはそのまま残り、undo 履歴は最後に正常保存された時点のものを使用する

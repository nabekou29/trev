# Feature Specification: 設定体系の整備

**Feature Branch**: `017-config-refinement`
**Created**: 2026-02-22
**Status**: Draft
**Input**: User description: "設定周りを整える。メニュー内にあるようなものも直接キーバインドを設定できるようにする。メニューの定義を設定でできるようにする。設定の命名を客観的に見直して、わかりやすいものとする。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - メニュー操作をショートカットキーで直接実行する (Priority: P1)

ユーザーはメニューを開かずに、メニュー内の個別アクションを直接キー操作で実行したい。例えば、ソート順を「名前順」に変更するために毎回ソートメニューを開くのではなく、一発のキー操作で切り替えたい。同様に、クリップボードへのコピーもメニューを経由せず直接実行したい。

**Why this priority**: メニュー操作のショートカット化はユーザーの操作効率に最も直結する改善であり、パワーユーザーの操作速度を大幅に向上させる。

**Independent Test**: メニュー内の個別アクションに対応するキーバインドを設定ファイルに記述し、メニューを開かずにそのキーを押すだけで目的のアクションが実行されることを確認できる。

**Acceptance Scenarios**:

1. **Given** デフォルト設定の状態、**When** ソートメニュー内の「名前順ソート」に対応するアクションにキーを割り当てる、**Then** そのキーを押すだけでソート順が名前順に切り替わる（メニューを開く必要がない）
2. **Given** デフォルト設定の状態、**When** コピーメニュー内の「絶対パスコピー」に対応するアクションにキーを割り当てる、**Then** そのキーを押すだけで絶対パスがクリップボードにコピーされる
3. **Given** メニュー内のアクションに直接キーバインドが設定されている状態、**When** 従来通りメニューを開いて同じアクションを選択する、**Then** 同じ結果が得られる（メニュー経由の操作も引き続き利用可能）

---

### User Story 2 - メニューを自由に定義する (Priority: P2)

ユーザーは設定ファイルで独自のメニューを自由に定義したい。例えば、よく使うアクションをまとめた「お気に入りメニュー」を作成してキーに割り当てたり、既存のコピーメニューが好みに合わなければ自分好みのコピーメニューを別途定義したりしたい。

**Why this priority**: メニューの自由定義はユーザーのワークフローに合わせた柔軟なカスタマイズを可能にする。P1の「直接キーバインド」があれば最低限の操作効率は確保されるため、優先度はP2とする。

**Independent Test**: 設定ファイルで独自のメニューを定義し、キーバインドに割り当て、アプリケーション起動後にそのキーを押してメニューが表示・動作することを確認できる。

**Acceptance Scenarios**:

1. **Given** 設定ファイルに独自メニューを定義しキーバインドに割り当てた状態、**When** そのキーを押す、**Then** 定義したメニューが表示され、各項目を選択すると対応するアクションが実行される
2. **Given** 複数の独自メニューを定義した状態、**When** それぞれのキーを押す、**Then** 各メニューが正しく表示される
3. **Given** メニュー定義を一切行っていない状態、**When** 組み込みメニュー（ソート・コピー）を開く、**Then** デフォルトの項目が従来通り表示される
4. **Given** 組み込みメニューと同じアクションを含む独自メニューを定義した状態、**When** 独自メニューから実行する、**Then** 組み込みメニューと同じ結果が得られる

---

### User Story 3 - わかりやすい設定項目名で設定を記述する (Priority: P3)

ユーザーは設定ファイルを編集する際に、各設定項目の意味を名前から直感的に理解したい。一貫した命名規則で、初めて設定を行うユーザーでもドキュメントを参照する頻度を減らしたい。

**Why this priority**: 命名の見直しは既存の動作を変えないリファクタリング的な改善であり、ユーザーの設定体験を向上させるが、機能的な拡張であるP1・P2より優先度は低い。

**Independent Test**: リネーム後の設定ファイルを用いてアプリケーションを起動し、すべての設定が正しく適用されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 新しい命名規則に基づく設定ファイル、**When** アプリケーションを起動する、**Then** すべての設定が正しく適用される
2. **Given** 旧命名規則で記述された設定ファイル、**When** アプリケーションを起動する、**Then** 不明な設定項目としてエラーまたは無視される（旧名称はサポートしない）

---

### Edge Cases

- メニュー内アクションに直接キーバインドを設定した上で、同じアクションがメニュー内にも残っている場合、両方のパスで正しく動作するか
- メニュー定義で項目を0個にした場合（空メニュー）、メニューを開くアクションは何も表示しないか無視するか
- ショートカットキーが重複するメニュー項目を定義した場合、最初の項目が優先されるか警告が出るか
- 旧名称の設定を使用した場合、パースエラーとなるか黙って無視されるか
- メニューのカスタム項目で定義したアクションが存在しないアクション名の場合、どのように扱われるか
- 設定ファイルの JSON Schema が新しい命名規則を反映し、エディタの補完・バリデーションが正しく機能するか

## Requirements *(mandatory)*

### Functional Requirements

**メニューアクションの直接キーバインド対応**

- **FR-001**: ソートメニュー内の各ソート順（名前・サイズ・更新日時・種類・拡張子・スマート）を、個別のアクション（`tree.sort.by_name` 等のドット階層形式）としてキーバインドに割り当てられなければならない
- **FR-002**: コピーメニュー内の各コピー対象（絶対パス・相対パス・ファイル名・拡張子なしファイル名・親ディレクトリパス）を、個別のアクション（`file_op.copy.absolute_path` 等のドット階層形式）としてキーバインドに割り当てられなければならない
- **FR-003**: 直接キーバインドで実行した場合も、メニュー経由で実行した場合と同じ結果が得られなければならない

**メニュー定義のカスタマイズ**

- **FR-004**: ユーザーは設定ファイルで任意のメニューを新規に定義できなければならない
- **FR-005**: メニュー項目にはタイトル、表示ラベル、ショートカットキー、実行するアクションを指定できなければならない。アクションとして組み込みアクション・シェルコマンド（`run:`）・IPC通知（`notify:`）の3種類すべてを使用できなければならない
- **FR-006**: メニュー項目の表示順序は設定ファイルでの定義順に従わなければならない
- **FR-007**: 組み込みメニュー（ソート・コピー）は、ユーザーがメニュー定義を行わなくても従来通り利用できなければならない
- **FR-008**: ユーザーが定義したメニューをキーバインドで呼び出せなければならない

**設定の命名規則見直し**

- **FR-009**: すべての設定項目名は一貫した命名規則（`snake_case`）に従わなければならない
- **FR-010**: 設定項目名は、その設定が何を制御するかを名前から推測できる程度に説明的でなければならない
- **FR-011**: 設定項目名およびアクション名の両方を見直し対象とし、類似の項目には一貫した接頭辞・接尾辞パターンを使用しなければならない（例：表示切替は統一された命名パターン、アクション名はドット階層形式で統一）
- **FR-012**: JSON Schema は新しい命名規則を反映し、エディタでの補完・検証に対応しなければならない
- **FR-013**: 既存のメニュー操作は、カスタマイズしていないユーザーにとって変更なく利用できなければならない

### Key Entities

- **アクション**: ユーザーが実行できる操作の単位。キーバインドやメニュー項目から参照される。ツリー操作・プレビュー操作・ファイル操作・ソート操作・コピー操作など多岐にわたる
- **キーバインド**: キー入力とアクションの対応関係。コンテキスト（ユニバーサル・ファイル・ディレクトリ）ごとに定義される
- **メニュー定義**: メニューに表示する項目の集合。各項目はラベル・ショートカットキー・アクション参照を持つ
- **設定項目**: アプリケーションの動作を制御するパラメータ。名前・値・デフォルト値・説明を持つ

## Clarifications

### Session 2026-02-22

- Q: メニューカスタマイズの上書き戦略は？ → A: メニューは自由に新規定義できる。既存メニューが好みでなければ別途定義してよい。既存メニューの部分修正ではなく、メニュー定義そのものをユーザーが自由に作成できる方式とする。
- Q: 新しい個別アクションの命名パターンは？ → A: `tree.sort.by_name`, `file_op.copy.absolute_path` のようにドット区切りの階層方式。既存カテゴリ（`tree`, `file_op`）を維持しつつ、サブグループで細分化する。
- Q: 非推奨（deprecated）警告の表示方法は？ → A: 旧名称のサポートは不要。リリース前のため破壊的変更を許容し、旧名称は無視して新名称のみをサポートする。
- Q: 設定名称リネームのスコープは？ → A: 全面見直し。設定項目名（config.yml）とアクション名の両方を対象に命名を見直す。
- Q: ユーザー定義メニューで使えるアクション種別は？ → A: 全種別対応。組み込みアクション・シェルコマンド（`run:`）・IPC通知（`notify:`）の3種類すべてをメニュー項目で使える。キーバインドと同じ能力を持つ。
- Q: 隠しファイル・gitignored ファイルの表示切替アクションのカテゴリは？ → A: `filter.*` カテゴリに分離する（`filter.hidden`, `filter.ignored`）。将来的に `filter.by_min_lines` 等のフィルター条件を追加可能な設計とする。`tree.*` からは移動する。

## Assumptions

- リリース前のため、旧設定名称の後方互換性は不要。破壊的変更を許容する
- メニューはユーザーが自由に新規定義可能とする。組み込みメニュー（ソート・コピー）はデフォルトとして残るが、ユーザーは代替メニューを自由に作成できる
- 設定ファイルのフォーマット（YAML）自体は変更しない
- メニュー内のキーナビゲーション（j/k/Enter/Esc）のカスタマイズは本スコープ外とする

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: メニュー内の全アクション（ソート6種 + コピー5種 = 11アクション）が個別にキーバインド可能になっている
- **SC-002**: メニューの項目定義を設定ファイルで変更でき、変更が即座にアプリ起動時に反映される
- **SC-003**: 新名称の設定で初めて設定ファイルを記述するユーザーが、ドキュメントを参照せずに設定項目の意味を80%以上正しく推測できる
- **SC-004**: 既存のデフォルトキーバインド・メニュー操作が、カスタマイズしていないユーザーにとって一切変更なく動作する
